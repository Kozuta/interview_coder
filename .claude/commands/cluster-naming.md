# /cluster-naming — Проверка и улучшение названий affinity-кластеров

Ты помогаешь UX/CX-исследователю переименовать кластеры аффинной диаграммы по принципу «название = инсайт».

## Что делать

Прочитай список кластеров, который передаёт пользователь (из файла, вставки или observations.json).
Для каждого названия:
1. Оцени по шкале: **Ярлык** / **Наблюдение** / **Инсайт**
2. Если это ярлык или слабое наблюдение — предложи улучшенный вариант

## Правила хорошего названия

**❌ Слабо** — просто ярлык, не двигает анализ:
- «Навигация», «Проблемы с онбордингом», «Обратная связь», «Работа с клиентами»

**✅ Сильно** — передаёт наблюдение, напряжение или поведенческий паттерн:
- «Пользователи теряются после первого входа, потому что не понимают, что делать дальше»
- «Доверие рушится, когда система не подтверждает действие»
- «Обходят основной флоу, чтобы сэкономить время»

**Тест:** если в название можно поставить «потому что» или «когда» — оно работает.

**Форматы:**
| Формат | Пример |
|---|---|
| Наблюдение + контекст | «Сравнивают цены вне продукта, потому что внутри нет фильтров» |
| Напряжение / противоречие | «Хотят кастомизацию, но боятся сломать настройки» |
| Поведенческий паттерн | «Первый шаг всегда — поиск, не меню» |
| JTBD | «Нужно быстро убедиться, что всё прошло» |

**Правила:**
- Один кластер = одна идея; если есть «и» — вероятно, это два кластера
- Пиши о пользователе, не о системе
- Избегай глагола «использовать» — он ничего не говорит о смысле
- Название — утверждение, с которым можно поспорить

## Формат ответа

Для каждого кластера:
```
Было: «Лояльность к отменам»
Оценка: Ярлык
Стало: «Спокойно принимает отмены записей, не отслеживая причины — клиентская база нестабильна»
```

Если название уже хорошее — скажи почему и оставь как есть.
В конце дай краткое резюме: сколько ярлыков, наблюдений, инсайтов.
