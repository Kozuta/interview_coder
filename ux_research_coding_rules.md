# UX Research Coding Rules

# Цель системы

Главная цель кодировки:

не красивый текст,
а качественный structured evidence dataset.

Приоритет:

- consistency > prose
- structure > interpretation
- evidence > storytelling

---

# Основной принцип

Каждая строка таблицы должна содержать:

- один наблюдаемый факт;
- одно действие;
- одну проблему;
- одну установку;
- один behavioral signal.

Одна строка = один смысл.

---

# Что такое atomic observation

Atomic observation — это:

- короткое;
- проверяемое;
- evidence-based описание.

Atomic observation должен:

- напрямую подтверждаться цитатой;
- содержать максимум 15–20 слов;
- не содержать аналитических выводов.

---

# Запрещено

Никогда не:

- обобщать пользователей;
- писать population-level statements;
- использовать формулировки «пользователи обычно», «бизнесы делают», «мастера предпочитают»;
- делать выводы;
- объяснять скрытые причины поведения;
- писать рекомендации;
- превращать observation в insight.

---

# Observation ≠ Insight

## Observation

Что пользователь сказал или сделал.

## Insight

Обобщение across users.

На этапе кодировки создаются только:

- observations;
- behavioral coding;
- clusters.

---

# Правила атомизации

Разделяй высказывание на несколько атомов, если:

- есть несколько действий;
- есть действие + причина;
- есть проблема + workaround;
- есть оценка + действие;
- есть проблема + эмоциональная реакция.

---

# Пример атомизации

Исходная цитата:

> «Фильтры постоянно сбрасываются, поэтому каждый раз заново всё настраиваю»

Нужно создать два атома:

1. «Фильтры сбрасываются после возврата»
2. «Каждый раз заново настраивает фильтры»

---

# Разрешённые behavioral codes

- Задача
- Триггер
- Паттерн поведения
- Драйвер
- Проблема
- Барьер
- Ценность
- Незнание возможности
- Недоверие / сомнение

---

# Правила для кода «Барьер»

Используй «Барьер» только если:

- пользователь отказался от действия;
- пользователь избегает сценария;
- пользователь прекратил выполнение задачи;
- задача осталась незавершённой.

Если пользователь всё равно выполнил задачу — это «Проблема».

---

# Модальности

Используй только:

- Необходимость
- Невозможность
- Возможность
- Уверенность
- Сомнение
- Оценка+
- Оценка−
- Норма
- Желательность
- Гипотетичность

---

# Правила модальности

Модальность должна опираться только на явные речевые маркеры.

## Примеры

| Маркер | Модальность |
|---|---|
| «приходится» | Необходимость |
| «не получается» | Невозможность |
| «удобно» | Оценка+ |
| «раздражает» | Оценка− |
| «наверное» | Сомнение |

Не придумывай модальность без явного речевого сигнала.

---

# Правила кластеризации

Cluster должен описывать:

- конкретную проблему;
- конкретный паттерн;
- конкретный workaround;
- конкретный driver;
- конкретное поведение.

## Плохие кластеры

- «Маркетинг»
- «Инструменты»
- «Бизнес-процессы»

## Хорошие кластеры

- «Опора на сарафанное радио»
- «Избегают найма команды ради гибкости»
- «CRM используется только как журнал записи»
- «Повторно настраивают фильтры после возврата»

---

# Правила форматирования

Используй:

- короткие предложения;
- простые конструкции;
- минимальный свободный текст;
- единый словарь терминов.

Не используй:

- длинные объяснения;
- narrative prose;
- аналитические эссе.

---

# Формат output

| respondent_id | raw_quote | atomic_observation | primary_code | secondary_code | modality | domain | theme | cluster |
|---|---|---|---|---|---|---|---|---|

---

# Критерии хорошей кодировки

Хорошая кодировка:

- короткая;
- конкретная;
- проверяемая;
- легко кластеризуется;
- не содержит инсайтов;
- не содержит generalized claims;
- пригодна для frequency analysis;
- пригодна для AI clustering.

---

# Пример хорошей кодировки

Исходная цитата:

> «У меня нет вывески, поэтому люди приходят только по рекомендациям»

Результат:

| atomic_observation | primary_code | modality | cluster |
|---|---|---|---|
| Нет вывески на улице | Проблема | — | Отсутствие офлайн-трафика |
| Клиенты приходят по рекомендациям | Паттерн поведения | Норма | Опора на сарафанное радио |
